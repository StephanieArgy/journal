Second Week: Monday, March 9, 2015

Github Hell

Second meeting...we walk in, innocently expecting a nice class about HTML and CSS and friendly website-building tools, and instead, we're confronted with three solid hours of Git! Aughh!

And yet...despite a few snafus (like my getting ahead of myself and accidently turning my entire Documents folder into a repository by typing git init before I should), the evening goes well. I feel as though I'm making small tenuous steps, but they all seem to be in a forward direction, so that's good.

The most important workflow, at least for now, seems to be this:

(Once we have our repositiory set up, on the local drive and remotely:)

  1. git status - To see what files have to be updated.
  2. If there are some, then git add to put them into the staging area.
  3. git commit the-file-we're-committing -m"Reason why we're doing the commit" to commit the changes and update the repository.
  4. git merge
  5. Then...to put the files on github, git push
    6. .

However...I will say that I'm not confident with the order of things, and I'm shaky on the way to specify who merges with whom, and whether I merge locally, or on the remote repository, or both.

Also, there was this swell little cheatsheet, which if it were on paper would now be all tattered and dog-eared from being used so much:

Workflow

Do your work on your branch. When you are ready to share your work with the team, do the following steps:

  • switch to the master branch ("git checkout master")
  • pull from github to bring in your team's work ("git pull")
  • switch to your branch ("git checkout mybranch")
  • merge from master to your branch ("git merge master")
  • resolve any merge conflicts that may happen using your editor
  • switch to the master branch ("git checkout master")
  • merge from your branch to master ("git merge mybranch")
  • push your changes to github ("git push")

Best. Thing. Ever.
