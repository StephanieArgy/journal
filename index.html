Basic Tools

In each segment of the Web Foundations course, we used a different set of tools and languages. In Primer Basic we used:

  • Slack - For communicating with our classmates
  • Chrome - The web browser of choice for our class, mainly because of its development tools
  • Codepen.io - A digital playpen for trying out code
  • Git - A version tracking system, which manages and stores all the revisions to a project.
  • Github.com - A service that hosts Git repositories online and also provides collaborative tools for code sharing and publication.
  • A text editor. PCS recommended Sublime, or we could try the open-source Atom, if we wanted to experiment with something new. (I chose Atom.)
  • Command line tools in Terminal. (All our Git and Github work on our local drives was done in the command-line -- which was terrifying initially, but gradually became more familiar.)

In Primer Pro, we used all of the above, and added:

  • JavaScript - The scripting language for Web pages
  • jQuery - A Javascript library that makes it much easier to use JavaScript on a website by creating "methods" for common tasks, so that those tasks can be used a single line of code that calls that method.

    • And in the third section, Advanced Web Dev Tools, we used:

    • Sass - A CSS pre-processor that makes it possible to create more readable style sheets with variables, reusable blocks of style patterns and other features, all which are then translated into CSS.
    • Node.js -- A platform of JavaScript based tools that can be used to write scalable network-based applications, in addition to JavaScript's more traditional browser-based uses.
    • npm - Installed along with node-js, npm is a package manager, which provides access to and manages over 125,000 pre-processors, scaffoldings and other packages, including three we used in this class: node-sass, Yeoman and Grunt.
    • Yeoman - An architecture and workflow used to build modern web apps, with three main tools: yo (for scaffolding an app), build (for building, previewing and testing an app) and the package manager (such as npm or Bower).
    • Grunt - A "task runner," used to automate certain repetitive tasks during the app-building process, including compilation, linting, unit testing, etc.
    • JSON - "JavaScript Object Notation." A syntax for the exchange of data between a webpage and a server. Easier to use than XML.
    • Selenium - A tool that installs as a plug-in on Firefox and lets developers create test cases to examine how user-interface elements are working on web applications.
    • AJAX - "Asynchronous JavaScript and XML." A way to use existing standards to exchange information with a server and update parts of a webpage without having to reload the entire thing.

Git and Github

One of the first topics in Primer Basic was the use of Git and Github -- Git for tracking iterations of a project in "repositories", Github for storing a copy of our repositories online.

In Primer Pro, our first JavaScript/jQuery assignment started with our having to make a repository and establish a remote on github.com. "This should be review," said the instructions.

So here's my review -- a knucklehead cheatsheet that I made for myself (and kept returning to regularly):

To Create a New Project:

  1. On GitHub, make a new repository.
  2. In Terminal, navigate to the directory that will contain the local directory for that respository.
  3. In that directory: git clone https://github.com/MyGitHubName/repositoryname.git.
  4. In Terminal, go into newly created local repo clone using cd.
  5. Back on GitHub, create a new branch called gh-pages.
  6. In Terminal (and in the new directory): git fetch origin gh-pages. (At this point, only master branch will be visible with git branch.)
  7. In terminal: git checkout gh-pages. (Now git branch will show both branches.)
  8. To start working -- in Terminal: git checkout master.

After working, to add changes and update gh-pages:

  1. In terminal, check status, see what's changed: git status.
  2. Add changes: git add (followed by files that have changed or added, one git add command for each). This puts the files into the staging area, so that they can be committed.
  3. Commit changes: git commit -m"Message describing this set of changes"
  4. Push changes to master branch online: git push.
  5. Change to gh-pages branch: git checkout gh-pages.
  6. Merge content from master branch: git merge master.
  7. Update online version of gh-pages: git push.

Also, in Primer Basic, we did our group project, and the assignment description included this swell little cheatsheet on using Github with a team. If this were on paper, it would be all tattered and dog-eared from having been used so much. In our repository, we had a master branch and a gh-pages branch, and each of also had our own personal branch on our local drive, where we would do our individual work. Then, to put it all together:

Workflow

Do your work on your branch. When you are ready to share your work with the team, do the following steps:

  • switch to the master branch ("git checkout master")
  • pull from github to bring in your team's work ("git pull")
  • switch to your branch ("git checkout mybranch")
  • merge from master to your branch ("git merge master")
  • resolve any merge conflicts that may happen using your editor
  • switch to the master branch ("git checkout master")
  • merge from your branch to master ("git merge mybranch")
  • push your changes to github ("git push")

Best. Thing. Ever.
